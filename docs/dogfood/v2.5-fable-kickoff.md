## Amend mechanics (v2 — mandatory this wave)

For **every** finding you fix:

1. **Test-first gate.** Write the failing test that probes the *full named invariant*. Run it. Confirm it FAILS against current HEAD. Only then fix. Re-run; confirm PASS. Report the test id + file:line.
2. **Meta-test where the finding is a vacuous mechanism** (C1, D1, D2, G2): the test must **mutate the protected thing and assert the mechanism fires** — delete the documented event → the template test goes RED; a factionless pack → the rubric dimension FAILS; a registered effect → it actually EXECUTES. "Passes N/N" ≠ "works." A gate you can't make go red is still vacuous.
3. **Family-of-call-sites probe.** After fixing the named site, grep every sibling of the pattern and fix the family (e.g. M2: every offensive resolver, not just `all-enemies`; A5: every public method building a path from `hash`).
4. **Build-green barrier.** `npm run build && npm run test` must pass after **each wave** (not just at the end). Run the suite 3× if any test looks flaky; record stability.
5. **Determinism guardrail** (applies to all `modules`/`core` fixes): no `Date.now`/`Math.random` in resolution; ticks from the engine counter; RNG only via seeded `world.rng`; all new ids via `genId(world, prefix)` (never the deprecated global `nextId`); every aggregation/selection ties-breaks by a total stable key.

### HIGH — close all six

**C1 — wire `applyEffects` (dead extension API).** `core/src/modules.ts:97` `ModuleManager.applyEffects` has zero callers; rule *checks* are dispatched in `engine.ts` but rule *effects* never run.
- **Invariant:** a module that registers a rule effect via `ModuleRegistrationContext.rules.registerEffect` has that effect **execute** during action dispatch, emitting through the normal event pipeline (ids via `genId`, deterministic).
- **Test-first:** register a test module whose rule effect flips a world flag / appends an event on a given verb; submit the triggering action; assert the effect ran. Confirm it FAILS today.
- **Fix (recommended arm):** call `applyEffects` in the dispatch pipeline *after* `checkRules`, threading results through `recordEvent` so ordering + ids stay deterministic.
- **⚠ ANDON:** if wiring alters event ordering or the id counter in a way that perturbs the byte-identical-replay tests, **STOP** — do not force it. Report the exact conflict. Fallback arm (advisor will decide): deprecate `registerEffect` with a runtime warning + doc note instead of wiring. Either way the outcome must be: the documented API is *honest* (works, or is clearly marked not-yet-supported).

**M1 — status trigger `chainDepth` accumulation.** `modules/src/status-effects.ts:369` seeds each damage event's proc queue with the accumulated `procCtx.chainDepth`; `:388` raises a shared high-water mark. Across an AoE's multiple damage events, later entities' thorns/reflect fire fewer times (or none once pinned at `PROC_DEPTH_LIMIT`).
- **Invariant:** identical reactive-trigger statuses on N entities hit by the same AoE in the same tick produce **identical reflect/thorns counts regardless of entity processing order**; the depth cap still prevents ping-pong within a single chain.
- **Test-first:** two defenders with identical spiked-armor reflect, one enemy AoE; assert both reflect the same number of times AND swapping their entity-id order changes nothing. Confirm FAILS today.
- **Fix:** seed each damage event's queue at `depth: 0`; keep the **shared** already-fired dedup `Set` (so a genuine ping-pong across events still terminates). Do not reset the dedup set per event.

**M2 — faction friendly-fire on the default target path.** `modules/src/ability-core.ts` `resolveTargets` `all-enemies`/`single` are type-only; `ability-intent.ts:315` offensive scoring too. `faction` is only consulted on the `effect.target:'zone'` path, but `effect.target` defaults to `'target'`.
- **Invariant:** an offensive ability (single or AoE) authored with the **default** `effect.target` never selects a target whose `affiliationOf(source, candidate)` is `ally`; a recruited same-faction, different-`type` ally is spared.
- **Test-first:** enemy casts an `all-enemies` AoE in a zone containing a same-faction, different-`type` recruited ally of the caster; assert the ally is NOT in the resolved target set (and a true enemy IS). Confirm FAILS today.
- **Fix:** route the `all-enemies`/`single`/offensive-scoring resolvers through the same `affiliationOf` predicate the zone path uses. **Family probe:** audit every offensive selection site (combat `combat-intent.ts:146` basic-attack = finding M3 — fold it in here for coherence, or defer M3 with rationale). Reconcile the v2.4 CHANGELOG line "enemy AoE spares allies" with the now-complete behavior.

**A1 — image-gen ComfyUI hang/crash.** `image-gen/src/comfyui-provider.ts` — queue POST (:107), history poll (:130), image fetch (:155) have no `AbortSignal`; `generate()` has no try/catch; `.json()` casts (:118,:133) throw on non-JSON. Mirror the correct pattern already in `ollama/src/client.ts`.
- **Invariant:** on a ComfyUI that accepts the connection then stalls, or returns a non-JSON 200 / 500, `generate()` **returns a typed failure within a bounded timeout** — never hangs, never throws raw.
- **Test-first:** a mock server for (a) stall past timeout, (b) non-JSON 200, (c) 500. Assert bounded typed failure each. (This is the first `comfyui-provider.test.ts` — highest-risk untested file in the repo.)
- **Fix:** thread `AbortSignal.timeout(this.opts.timeoutMs ?? 120_000)` into all three fetches; wrap `generate()` so network/`.json()` throws become a typed `{ok:false,error}`; cap `arrayBuffer()` size and validate `content-type` starts with `image/` (folds in A6).

**A5 — asset-registry path traversal.** `asset-registry/src/file-store.ts` `get/getMeta/has/delete` build paths from raw `hash`; `isValidHash` ships in `hash.ts` but is never called at these sites.
- **Invariant:** every public method rejects a `hash` not matching `/^[a-f0-9]{64}$/` (return null / no-op / typed error) — no path escapes the store root.
- **Test-first:** `delete('../../x')` no-ops and unlinks nothing outside the store; `get('../../etc/x')` returns null. Confirm the delete FAILS today (unlinks outside). **Family probe:** all four methods.
- **Fix:** `isValidHash(hash)` guard at the top of each.

**A4 — content-addressing integrity on read.** `file-store.ts:57-64` `get()` returns bytes without re-hashing.
- **Invariant:** `get(hash, {verify:true})` returns null / typed integrity error when `sha256(bytes) !== hash`.
- **Test-first:** tamper a stored `.bin`, assert `get({verify:true})` detects it. **Fix:** optional re-hash behind a `verify` flag (default off for hot paths, documented).

**D1 — pack rubric checks the wrong field.** `pack-registry/src/rubric.ts:60-70` `checkDistinctFactionTopology` tests `defaultModules.includes('dialogue-core')` and returns the false detail "Pack defines dialogue and faction structures."
- **Invariant:** the dimension passes iff the pack actually defines districts with controlling factions; the detail string reflects what was inspected.
- **Test-first (meta):** a pack with `dialogue-core` but **zero factions** must FAIL this dimension (today it passes). **Fix:** inspect real faction/district data; correct the detail string.

**D6 — rubric uniqueness dims only check existence.** Same file: `checkDistinctVerbs` passes on `uniqueVerbs.length>=1` (not `trulyUnique`); `checkDistinctFailureMode` never uses its `others` param.
- **Invariant:** dimensions the doc calls "cross-catalog unique" fail when another pack shares the trait. **Fix:** use `others`, or soften `PACK_RUBRIC.md` to "present" for those dims (advisor prefers *fix*, since the doc sells uniqueness).
